# アノテーション: ルーブリック・サブエージェント分担・マージ

エージェント（Claude）自身が Read ツールで画像を見てラベルを付ける。
**収集バケット名からラベルを推定してはならない**。バケットは検索の網にすぎず、
「dog_photo バケットに犬が写っていない画像」は普通に混入する。全画像を
ルーブリックに基づいて独立に判定すること。

## ルーブリック（`data/annotation/rubric.md`）

Phase 3 の最初に書く。サブエージェントへの唯一の判定基準書になるので、
このファイルだけ読めば判定できる自己完結した内容にする。構成:

```markdown
# アノテーションルーブリック: {タスク名}

## タスク
{config.yaml の task.description と同じ内容}

## ラベル定義（分類の場合）
### {class_1}
- 含める: ...（具体的な視覚条件）
- 含めない: ...（隣接クラスとの境界）
### {class_2}
...

## 境界判定の順序
迷ったら次の順で判定する: {例: 1. 対象が写っているか → 2. 主要被写体か → ...}

## DISCARD（除外）基準
以下は DISCARD とし、理由を1行で書く:
- タスクと無関係な被写体
- 画像が壊れている・判読不能
- クリップアート・イラスト・CG（実写タスクの場合）
- どのクラス定義にも該当しない

## 判定例（チェックポイント②の後に追記）
- {filename}: {label} — {理由}
```

**回帰タスクの場合**、ラベル定義の代わりに値アンカーを書く:

```markdown
## 値アンカー（{min}〜{max} {unit}）
- {min}: ...（この値になる画像の具体的な見た目）
- {中間値}: ...
- {max}: ...
丸め: 0.5 刻み。アンカー間は補間して判定。採点不能な画像は DISCARD。
```

## サンプルアノテーション（Phase 3・チェックポイント②）

1. バケット横断で10〜15枚を選ぶ（各バケットから均等に）
2. **メインセッションで** 各画像を Read し判定
3. ユーザーに markdown 表で提示: `| filename | バケット | ラベル | 理由 | 確信度 |`
   （ユーザーが画像を確認できるようファイルパスも示す）
4. フィードバックを受けたら rubric.md を更新（境界ルールの明文化、few-shot 例の追記）
5. 判定が大きくズレていた場合はサンプルをやり直して再提示。承認を得てから Phase 4 へ

## 全量アノテーション（Phase 4）

### バッチ分割と冪等性

1. `data/annotation/batches/` の既存 `batch_*.csv` を読み、処理済み filename の集合を作る
2. `data/collection_metadata.csv` の全画像から処理済みを除いた残りを **25枚ずつ** のバッチに分割
3. バッチ番号は既存の続きから採番（`batch_01.csv`, `batch_02.csv`, ...）

### サブエージェント起動

general-purpose サブエージェントを **2〜4並列** で起動。プロンプト雛形:

```
あなたは画像アノテーターです。

1. {絶対パス}/data/annotation/rubric.md を Read して判定基準を頭に入れる
2. 以下の画像を1枚ずつ Read し、ルーブリックに従ってラベルを判定する:
   {絶対パスのリスト、~25枚}
3. 結果を {絶対パス}/data/annotation/batches/batch_{NN}.csv に書く。
   ヘッダー: filename,label,confidence,reason
   - filename: 拡張子含むファイル名のみ（パスなし）
   - label: {クラス名リスト} のいずれか、または DISCARD
   - confidence: high / medium / low
   - reason: 判定理由を日本語で1行（カンマを含む場合は引用符で囲む）
4. 全行数と ラベル別件数を報告する

判定できない・ルーブリックに該当しない画像は無理にラベルを付けず DISCARD にすること。
```

### 検収

各サブエージェント完了後:
- バッチCSVの行数 = 依頼枚数 か確認。欠けたファイルだけ集めて追加バッチで再依頼
- `confidence=low` が3割を超えるバッチはルーブリックの曖昧さを疑う（ユーザーに報告）

### マージ

```bash
uv run python .claude/skills/image-ml-agent/scripts/merge_annotations.py --dry-run  # 検証のみ
uv run python .claude/skills/image-ml-agent/scripts/merge_annotations.py            # annotations.csv 生成
```

config.yaml のクラス/範囲に対してラベルを検証し、DISCARD を除外（理由の内訳を表示）、
filename 重複は後勝ち、`data/images/` に実在しないファイルはエラー。
**config.yaml が先に存在している必要がある**（Phase 0 で生成済みのはず）。

## クラス不均衡の処理（マージ後）

- **ハードフロア**: 全クラス ≥ 14枚（val/test 15%ずつの層化分割で各splitに2枚以上入る最低線）。
  下回るクラスがあれば追加収集が必須
- 少数クラス < max(15, 最多クラスの30%) の場合、ユーザーに選択肢を提示:
  - (a) 少数クラス狙いの追加収集ラウンド（プランJSONに専用バケットを足し、新規分のみアノテーション）
  - (b) このまま続行（LightGBM + macro-F1 は軽度の偏りには耐える）
- 追加収集した場合は Phase 4 の冪等ロジックがそのまま使える（新規ファイルだけがバッチ対象になる）
